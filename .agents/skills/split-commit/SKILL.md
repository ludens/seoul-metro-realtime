# 커밋 분리

## 개요

뒤섞인 작업 트리를 몇 개의 논리적인 커밋으로 정리합니다. 목표는 커밋 수를 줄이는 것이 아니라, 리뷰하기 쉽고 필요할 때 안전하게 되돌릴 수 있는 히스토리를 만드는 것입니다.

## 사용 시점

- 사용자가 하나의 변경을 여러 커밋으로 나눠 달라고 할 때
- 작업 트리에 서로 무관하거나 연관성이 약한 변경이 섞여 있을 때
- 생성 파일을 소스 변경과 분리해야 할 때
- 큰 기능을 하위 시스템별 중간 커밋으로 나눠 관리해야 할 때

## 핵심 원칙

각 커밋은 하나의 분명한 의도를 담아야 하며, diff와 커밋 메시지만 봐도 무엇을 위한 변경인지 이해할 수 있어야 합니다.

## 작업 순서

### 1. 작업 트리 확인

```bash
git status --porcelain
git diff --stat
git diff
```

이미 스테이징된 변경이 있다면, 어떻게 나눌지 결정하기 전에 staged diff와 unstaged diff를 모두 확인합니다.

```bash
git diff --staged
git diff
```

### 2. 커밋 그룹 정리

파일 확장자가 아니라 변경 의도를 기준으로 묶습니다. 좋은 분리 예시는 다음과 같습니다.

- 소스 로직
- 테스트
- 생성 결과물
- 문서
- 뒤이은 커밋에 필요한 선행 리팩터링

다음과 같은 방식은 피합니다.

- 하나의 기능을 함께 구현하는데도 프런트엔드 파일과 백엔드 파일을 억지로 나누는 것
- 파일 수만 맞추려는 식의 임의 분할
- 사용자가 원하지 않았는데도 강하게 결합된 코드와 테스트를 따로 떼는 것

### 3. 한 그룹씩 스테이징

가능하면 경로를 명시해서 스테이징합니다.

```bash
git add path/to/file1 path/to/file2
```

하나의 파일 안에 서로 다른 논리 단위의 변경이 섞여 있을 때만 patch 모드를 사용합니다.

```bash
git add -p path/to/file
```

한 그룹을 스테이징한 뒤에는 경계가 올바른지 확인합니다.

```bash
git diff --staged --stat
git diff --staged
```

### 4. 집중된 메시지로 커밋

Conventional Commits 형식을 사용합니다. 커밋 메시지는 지금 스테이징된 그룹에만 맞춰 작성합니다.

```bash
git commit -m "<type>[scope]: <description>"
```

### 5. 남은 그룹 반복 처리

커밋을 하나 만들 때마다 남은 변경 사항을 다시 확인합니다.

```bash
git status --porcelain
git diff --stat
```

## 분리 기준

- 사용자에게 보이는 새 기능이면 `feat`를 우선 사용합니다
- 동작 수정이면 `fix`를 우선 사용합니다
- 동작 변화 없는 코드 이동이나 정리라면 `refactor`를 우선 사용합니다
- 문서만 바뀌었다면 `docs`를 우선 사용합니다
- 테스트만 바뀌었다면 `test`를 우선 사용합니다
- 사용자에게 직접 드러나지 않는 생성 산출물이나 유지보수 작업이면 `chore`를 우선 사용합니다

## 권장 커밋 순서

순서를 여러 방식으로 잡을 수 있다면 보통 아래 순서를 따릅니다.

1. 선행 리팩터링
2. 핵심 기능 또는 버그 수정
3. 테스트
4. 문서
5. 생성 결과물

이 순서가 히스토리를 읽기 쉽고 bisect 하기에도 유리합니다.

## 안전 수칙

- 작업 트리를 깔끔하게 보이게 하려고 무관한 삭제를 억지로 스테이징하지 않습니다
- 이해하지 못한 사용자 변경을 내 커밋에 섞지 않습니다
- 사용자가 요청하지 않았다면 이전 커밋을 amend 하지 않습니다
- 커밋을 억지로 나누려고 파괴적인 git 명령을 쓰지 않습니다
- 어디서 나눠야 할지 애매하면 커밋하기 전에 제안하는 그룹을 먼저 설명합니다

## 출력 형식

커밋에 들어가기 전에 아래처럼 짧은 분리 계획을 먼저 제시합니다.

```text
1. feat(markdown): add PDF-to-markdown extraction script
2. feat(markdown): add markdown review status board
3. chore(output): add extracted card markdown outputs
```

그다음 각 그룹을 순서대로 스테이징하고 커밋하며, 매번 staged diff를 확인합니다.
